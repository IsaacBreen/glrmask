### 2. Root cause diagnosis (deep dive)

**The JS grammar's right-recursive cycle Structure:**

With `exhaustive`, the grammar has a 4-NT right-recursive cycle:
```
expression → assignment_expression → arrow_function → arrow_function_0 → expression
```

With chain strategies, additional `__nl_*` NTs extend this to a 10-NT cycle:
```
expression → assignment_expression → arrow_function → arrow_function_0 →
  conditional_expression → conditional_expression_0 → yield_expression →
  __nl_yield_expression_0 → __nl_yield_expression_1 → iteration_statement_1 → expression
```

The extra NTs come from chain encoding of nullable runs: e.g., `yield_expression` has a nullable run that gets encoded as `__nl_yield_expression_0 → __nl_yield_expression_1 → iteration_statement_1`.

**Why the baseline algorithm looped:**

`resolve_indirect_right_recursion` (baseline at `d8914e88d`) finds cycles via DFS and inlines `cycle[0] → cycle[1]`:

```rust
inline_right_end(productions, &cycle[0], &cycle[1], &nullable);
```

With `exhaustive`:
- DFS visits `assignment_expression` first (alphabetical order), finds cycle starting there.
- Inlines `arrow_function` INTO `assignment_expression`.
- `arrow_function` has no direct right recursion, so the inlined productions don't re-create the `assignment_expression → arrow_function` edge.
- Works in 14 iterations, 529 productions.

With `right_chain`:
- The `__nl_*` NTs (starting with `_`, alphabetically before `a`) are added to the graph.
- DFS starts from `__nl_*`, reaches `expression`, finds cycle starting with `expression`.
- Inlines `assignment_expression` INTO `expression`.
- **Critical**: `assignment_expression` has direct right recursion (`assignment_expression → ... assignment_expression`). When these self-recursive productions are inlined into `expression`, they become `expression → ... assignment_expression`, **re-creating the same edge**.
- The cycle `[expression, assignment_expression]` is found again next iteration. Infinite loop.

**Key insight**: Inlining a target NT that has direct right recursion re-creates the source→target edge, because the target's self-recursive productions become source→target productions.

### 3. Fix attempts that failed (important for posterity)

**Attempt 1: Call `resolve_direct_right_recursion` inside the indirect loop (full grammar)**
- After each inline, resolve all direct-RR across all NTs.
- Result: ALL strategies timeout including exhaustive. Creates infinite NT growth (`expression'`, `expression''`, `expression'''`, ...) because the full-grammar direct-RR pass creates new NTs that are then processed again.

**Attempt 2: Filter direct-RR from `inline_right_end` (skip `rhs.last() == from_nt`)**
- Added a filter to skip inlined productions that would create direct right recursion.
- Result: Exhaustive works (23s) but chain strategies timeout. The filter prevents `inline_right_end` from making progress—if all of `to_nt`'s productions end in `from_nt`, no inlining happens, and the same cycle is found every iteration.

**Attempt 3: Per-cycle ordered substitution (Aho-Ullman style)**
- For each cycle, fix an order on cycle NTs. For each NT_i, inline all earlier NTs NT_j (j < i) at right-end positions, then eliminate direct-RR on NT_i.
- Result for small cycles (2-6 NTs): works fine.
- Result for the 10-NT cycle: **grammar explosion**. Ordered substitution on 10 NTs means NT[9] must absorb productions of NTs[0..8]. Each inlining step multiplies productions combinatorially. The `compute_nullable_nonterminals` function was called repeatedly (~312ms each), indicating the grammar grew unboundedly within the ordered substitution of the 10-NT cycle.
- With GPT-2 vocab: all 4 strategies OOM-killed at ~43-45s (Killed: 9).
- With tiny vocab: found 8 cycles; process stuck in cycle 4 (the 10-NT one).

**Attempt 4: Edge-breaking with direct-RR elimination per-edge**
- Find cycle, pick cheapest edge, inline, resolve direct-RR for `from_nt` only.
- Result: exhaustive produces 63,801 prods (vs baseline 529). Each direct-RR elimination roughly doubles productions for the affected NT. With ~16 iterations, this compounds to >100x grammar growth. All strategies run but the grammar is so large that DFA construction takes >90s.

### 4. The working fix

**Strategy: prefer cycle edges where `to_nt` has NO direct right recursion.**

The key observation: if `to_nt` does NOT have direct right recursion (no production `to → ... to`), then inlining it into `from_nt`:
1. Replaces all `from → ... to` productions with `to`'s alternatives.
2. Since `to` doesn't self-recurse, the inlined alternatives don't end in `to`.
3. Therefore `from → to` is permanently removed from the right-reachability graph.
4. No direct-RR elimination needed → no grammar growth.

Implementation in `resolve_indirect_right_recursion`:

```rust
// Check which NTs in the cycle have direct right recursion
let has_direct_rr: Vec<bool> = cycle.iter().map(|nt| {
    productions.iter().any(|p| &p.lhs == nt && is_direct_right_recursive(p))
}).collect();

// Prefer an edge where to_nt has NO direct RR
let safe_edge = (0..cycle.len()).find(|&i| {
    let to_idx = (i + 1) % cycle.len();
    !has_direct_rr[to_idx]
});

if let Some(edge_idx) = safe_edge {
    // Safe: inline won't re-create the edge
    inline_right_end(productions, &cycle[edge_idx], &cycle[next], &nullable);
} else {
    // All targets have direct RR. Forced fallback: inline + direct-RR elimination.
    inline_right_end(productions, from, to, &nullable);
    resolve_direct_right_recursion_single_nt(productions, from, new_name_generator);
}
```

The fallback path (all targets self-recurse) was never triggered on the JS grammar.

## Technical Notes for Future Reference

### Why DFS order matters for `resolve_indirect_right_recursion`

The DFS cycle finder iterates over `BTreeMap` keys (alphabetical). If new NTs are added that come alphabetically before existing ones (like `__nl_*` < `assignment_expression`), the DFS traversal order changes. This means the "first cycle found" may start from a different NT, leading to a different inlining choice that may or may not make progress.

The fix makes the algorithm **order-independent**: regardless of which cycle is found first or which NT it starts with, the safe-edge selection ensures progress by choosing an edge where inlining permanently breaks the cycle.

### When the fallback (direct-RR elimination) would trigger

The fallback triggers only when EVERY NT in a cycle has direct right recursion. This would mean the cycle has the form:
```
A → ... A (self-loop) and A → ... B → ... A (cycle with B)
B → ... B (self-loop) and B → ... A → ... B (cycle with A)
```

In practice, most grammar NTs don't have direct right recursion. The JS grammar's right-recursive NTs (`assignment_expression`, `expression`) are a minority, and cycles always include non-self-recursive NTs like `arrow_function`, `conditional_expression`, etc.
