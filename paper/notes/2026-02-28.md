# 2026-02-28

## Objective

Develop torture-test JSON Schemas that expose performance weaknesses in constrained decoding engines (Sep1 and LLGuidance). Find small, focused schemas that cause disproportionately bad performance, particularly:
- LLG mask gen or commit >10ms
- LLG build time >10s
- Sep1 build time or mask gen pathologies

## Key Findings

### Finding 1: LLG is robust against small schemas

No small schema (<10KB) achieves >3ms mask gen on LLG. We tested 65+ schemas across 11 categories:
- Exponential regex patterns (`(a?){N}a{N}`) — 112 bytes → 0.18ms max (n=25)
- Recursive trees (198 bytes, depth 30) — 0.15ms max
- Deep late discriminators (1.3KB, depth 10, 5 branches) — 0.21ms max
- Overlapping string patterns (3.1KB, 20 patterns × 15 len) — 0.96ms max
- Nested oneOf with string discrimination (8KB, 64 leaves) — 0.10ms max
- Multi-property enum cross product (6.4KB, 5 props × 50 values) — 0.10ms max
- Confusable enum (same-length, diverge at end, 500 values) — 1.40ms max
- Shared-prefix enum (500 values, 60-char prefix) — 1.71ms max

LLG uses NFA simulation for regex terminals, avoiding DFA state explosion. Its Earley/GLL parsing handles structural complexity (recursion, nesting, branching) with near-constant cost per token.

### Finding 2: LLG's single weakness is string enum count

LLG mask gen scales linearly with the number of enum/const string values:

| Enum Values | Schema KB | LLG Build | LLG Max | LLG p99 | Sep1 Build | Sep1 Max |
|---|---|---|---|---|---|---|
| 500 | 17 | 0.013s | 1.40ms | 1.39ms | 0.46s | 0.03ms |
| 2000 | ~90 | 0.068s | 0.87ms | 0.87ms | 4.9s | 0.07ms |
| 3000 | ~130 | 0.080s | 1.79ms | 1.79ms | 8.1s | 0.22ms |
| 5000 | 306 | 0.215s | 9.76ms | 9.59ms | 19.0s | 7.74ms |
| 7000 | ~430 | 0.335s | 8.95ms | 8.80ms | 22.4s | 0.26ms |
| 10000 | 811 | 0.383s | **18.60ms** | **18.24ms** | 20.9s | 0.87ms |
| 15000 | ~1200 | 0.776s | **43.95ms** | **43.10ms** | 46.2s | 4.56ms |
| 20000 | 2021 | 1.412s | **79.44ms** | **77.90ms** | 51.4s | 3.84ms |

To breach 10ms mask gen on LLG requires ~5000+ enum values → 300KB+ schema. This is not an algorithmic weakness — it's linear scaling with grammar size.

### Finding 3: Sep1's weakness is DFA-exponential regex

The regex `[ab]*a[ab]{N}` in a string pattern creates 2^N DFA states:

| N | Schema Bytes | Sep1 Build | Sep1 Max | LLG Build | LLG Max |
|---|---|---|---|---|---|
| 10 | 148 | 1.17s | 0.20ms | 0.004s | 0.15ms |
| 15 | 148 | **50.95s** | 3.87ms | 0.004s | 0.65ms |
| 20 | 148 | killed | - | 0.004s | 0.16ms |
| 25 | 148 | killed | - | 0.004s | 0.18ms |

This is the most extreme cost/size ratio found: **148 bytes → 51s build**. LLG handles all sizes in <5ms because it uses NFA simulation.

Note: the similar regex `(a?){N}a{N}` does NOT trigger this in Sep1 because CFA normalizes it into grammar rules (optional literals), not a raw regex terminal.

### Finding 4: Sep1 struggles with structurally diverse oneOf

When oneOf branches have different *structures* (different numbers of required properties), Sep1's state merging optimization fails:

| Families | Variants/Fam | Total Vars | Schema KB | Sep1 Build | Sep1 Max | Sep1 p99 |
|---|---|---|---|---|---|---|
| 20 | 10 | 200 | ~50 | 4.1s | 2.17ms | 1.91ms |
| 50 | 20 | 1000 | ~300 | 43.2s | 1.33ms | 1.27ms |
| 100 | 5 | 500 | ~100 | 129.0s | 17.85ms | 12.24ms |
| 100 | 10 | 1000 | ~500 | 215s | **40.84ms** | **28.29ms** |

But these require large schemas (>50KB). Same-structure variants (flat cross-product) are handled efficiently by Sep1's state merging.

### Finding 5: LLG crashes on nested discriminated unions

The schema `nested_disc_d3_b15` (depth 3, branching 15, ~3.7MB) crashes LLG with:
```
assertion failed: grammar.symbols.len() < u16::MAX as usize - 10
```
LLG's symbol count is limited to ~65K, and deeply nested discriminated unions with high branching exceed this.

### Finding 6: K-level nested late discriminator achieves EXPONENTIAL parser stress

**This is the key new result.** A nested late discriminator with unique inner schemas at each level defeats GLL's Graph Structured Stack (GSS) sharing, creating B^K live parser states.

**Construction:** K levels of `oneOf`, branching factor B. Each branch is an array `[inner, disc]` where `disc` is a const string encoding the unique branch path. Because each branch path creates a distinct grammar rule, the GLL parser cannot share/merge states across branches.

Compact array-based (B=2):

| K | Effective (2^K) | Schema KB | mask_max | Status |
|---|---|---|---|---|
| 6 | 64 | 12 | 0.09ms | ✓ complete |
| 7 | 128 | 25 | 0.16ms | ✓ complete |
| 8 | 256 | 50 | 0.33ms | ✓ complete |
| 9 | 512 | 101 | 0.57ms | ✓ complete |
| 10 | 1024 | 205 | 1.33ms | ✓ complete |
| 11 | 2048 | 415 | 1.61ms | died step 6 |
| 12 | 4096 | 841 | **7.29ms** | died step 6 |

Compact array-based (B=3):

| K | Effective (3^K) | Schema KB | mask_max | Status |
|---|---|---|---|---|
| 4 | 81 | 12 | 0.11ms | ✓ complete |
| 5 | 243 | 35 | 0.29ms | ✓ complete |
| 6 | 729 | 106 | 0.88ms | ✓ complete |
| 7 | 2187 | 322 | 1.44ms | died step 4 |
| 8 | 6561 | 980 | **13.16ms** | died step 4 |

**Growth is exponential:** mask_max ≈ 0.0013 × B^K ms. Each level doubles (B=2) or triples (B=3) the mask cost.

**Hard ceiling:** LLG's grammar symbol limit (`u16::MAX ≈ 65K`) caps total unique grammar rules. With ~5 symbols per rule, max effective alternatives ≈ 65K/5 ≈ 13K. B=3 K=8 (6561 eff) is near this limit.

**Why it works:** Standard recursive schemas use `$ref` which creates SHARED grammar rules — GLL's GSS compresses these into O(1) state per recursion depth. The K-level discriminator avoids sharing by making each branch's inner schema structurally unique (different const values per branch path). The parser must track B^K concurrent parse positions.

**Death mechanism:** The schema→grammar converter (`schema_to_grammar_ir.py` → `to_llguidance_lark.py`) INLINES all B^K alternatives into a single grammar rule (`_arr_item_1`). This puts B^K items into one Earley "row", exceeding the `max_items_in_row: 2000` limit. Raising all LLParserLimits does NOT fix this — the Earley parser's per-row cost is inherently O(B^K).

### Finding 7: Direct Lark grammars with named sub-rules bypass parser death

**This is the key breakthrough.** By generating Lark grammars DIRECTLY (bypassing the schema→grammar converter), with **named sub-rules per level**, we distribute parser items across multiple rules. Each rule has at most B alternatives instead of B^K, completely eliminating the death issue.

**Construction:** For K=3, B=23, the grammar has rules like:
```
_lv3_root: _lv2_0 | _lv2_1 | ... | _lv2_22
_lv2_0: "[" _lv1_0_0 ","  "\"d_0\"" "," "\"l2_0\"" "]"
_lv2_1: "[" _lv1_1_0 ","  "\"d_1\"" "," "\"l2_1\"" "]"
...
_lv1_0_0: "[" _lv0_0_0_0 "," "\"d_0\"" "," "\"l1_0_0\"" "]"
...
_lv0_0_0_0: "\"leaf_0_0_0\""
```

Each `_lvN_*` rule has exactly B alternatives. The parser sees O(B) items per row, not O(B^K).

**Results — Direct Lark vs Schema approach:**

| B | K | Eff B^K | Approach | mask_max | Status |
|---|---|---------|----------|----------|--------|
| 42 | 2 | 1,764 | Schema | 2.27ms | ✓ complete (prior best) |
| 10 | 4 | 10,000 | Schema | 19.75ms | ✗ died step 2 |
| 13 | 3 | 2,197 | Schema | 1.17ms | ✗ died step 2 |
| 6 | 4 | 1,296 | DirectLark | 0.94ms | ✓ complete |
| 10 | 4 | 10,000 | DirectLark | 19.88ms | ✓ complete |
| 20 | 3 | 8,000 | DirectLark | 12.06ms | ✓ complete |
| 21 | 3 | 9,261 | DirectLark | 15.45ms | ✓ complete |
| 22 | 3 | 10,648 | DirectLark | 20.67ms | ✓ complete |
| **23** | **3** | **12,167** | **DirectLark** | **26.53ms** | **✓ complete (NEW CHAMPION)** |
| 24 | 3 | 13,824 | DirectLark | — | ✗ grammar symbol overflow |
| 6 | 5 | 7,776 | DirectLark | 13.59ms | ✓ complete |
| 4 | 6 | 4,096 | DirectLark | 4.47ms | ✓ complete |
| 3 | 8 | 6,561 | DirectLark | 13.55ms | ✓ complete |
| 2 | 12 | 4,096 | DirectLark | 7.06ms | ✓ complete |

**Key insight:** Every DirectLark grammar COMPLETES (up to the `u16::MAX` symbol limit). The schema-based approach DIES at B^K > ~1800 because the converter inlines everything into one rule. DirectLark distributes work across named sub-rules, pushing the hard limit to the grammar symbol ceiling (~65K symbols → B=23 K=3 with 12,167 effective alternatives).

**12× improvement:** The new champion (B=23 K=3 → 26.53ms completing) is 12× the prior best completing result (B=42 K=2 → 2.27ms with schema approach).

**Grammar symbol limit per K:**
- K=3: max B=23 (~63.6K symbols)
- K=4: max B=10 (~55.6K symbols)
- K=5: max B=6 (~54.4K symbols)
- K=6: max B=4 (~21.8K symbols)

**Optimization attempts evaluated:**
- Inlining level-0 rules into level-1: No improvement (B=24 K=3 still overflows)
- Merging terminal symbols (`","`, disc, `"]"` → one terminal): Tokenization mismatch errors

**Default vs raised limits:** B=6 K=4 works with DEFAULT LLParserLimits (mask_max 1.00ms). The named sub-rules approach avoids the `max_items_in_row` bottleneck entirely, so raised limits are only needed for the very largest configs.

## Conclusion

**LLG's GLL parser has a genuine exponential worst-case** via K-level nested late discriminators with unique inner schemas. The parser must track B^K concurrent parse positions, giving mask_max ≈ 0.0013 × B^K ms.

**The schema→grammar converter is the bottleneck, not the parser itself.** The converter inlines all alternatives into one rule, causing death at B^K > ~1800. Direct Lark grammars with named sub-rules (O(B) alternatives per rule) push the completing frontier to B^K = 12,167 (B=23 K=3 → 26.53ms mask_max).

The HARD limit is LLG's grammar symbol ceiling (`u16::MAX ≈ 65K`), not parser complexity. With named sub-rules, every grammar that fits within the symbol limit completes.

For LEXER stress, large string enums (≥5000 values, ≥300KB) achieve >10ms mask gen through linear scaling.

For Sep1, the pathological regex (`[ab]*a[ab]{15}`) is the only small-schema (<1KB) torture case that causes severe build-time regression (51s from 148 bytes).

## Schema Complexity Metrics

We defined 8 metrics for schema complexity: raw bytes, keyword count, oneOf/anyOf branch count, property count, enum value count, nesting depth, $ref count, pattern count.

Key insight: **schema bytes alone is the strongest predictor of LLG mask gen time**, because LLG scales linearly with grammar terminals. Other structural metrics (depth, branching, recursion) have negligible impact — EXCEPT for the K-level nested discriminator construction, where nesting depth K creates exponential parser cost.
