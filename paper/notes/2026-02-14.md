## Session 2 (continued, same day)

### 13) Attempted "fix" for token 34713 — commit-probe fallback (REVERTED)

The first attempt at fixing the token 34713 false negative was a brute-force safety net called `recover_missing_tokens_with_commit_probe()` (commit `74390f835`, **reverted** as `4e798ff96`).

**What it did:**
- After the DWA mask computation in `get_mask()`, it collected all candidate tokens from `possible_matches` not already in the mask.
- For each candidate, it *cloned the entire constraint state* and called `commit_bytes(token_bytes)`.
- If the cloned state remained `is_active()`, it inserted the token into the mask.

**Why it's terrible:**
- **It's cheating.** Instead of fixing the root cause (DWA under-approximation), it adds a brute-force probe that bypasses the entire precompute pipeline.
- **Performance disaster.** For every `get_mask()` call, it potentially clones the full parser state and runs `commit_bytes()` for hundreds of candidate tokens.
- **Architectural regression.** The whole point of the DWA precompute pipeline is to avoid per-token probing. This "fix" regresses to the naive approach exactly where it shouldn't.
- **It masks the real bug.** The underlying issue in the DWA weight computation or `possible_matches` construction still exists — this just papers over it.

**Lesson:** When the precompute pipeline under-approximates, the fix must be in the precompute pipeline, not a runtime fallback that defeats the purpose of precomputation.
