## Session 2 (continued, same day)

### 13) Attempted "fix" for token 34713 — commit-probe fallback (REVERTED)

The first attempt at fixing the token 34713 false negative was a brute-force safety net called `recover_missing_tokens_with_commit_probe()` (commit `74390f835`, **reverted** as `4e798ff96`).

**What it did:**
- After the DWA mask computation in `get_mask()`, it collected all candidate tokens from `possible_matches` not already in the mask.
- For each candidate, it *cloned the entire constraint state* and called `commit_bytes(token_bytes)`.
- If the cloned state remained `is_active()`, it inserted the token into the mask.

**Why it's terrible:**
- **It's cheating.** Instead of fixing the root cause (DWA under-approximation), it adds a brute-force probe that bypasses the entire precompute pipeline.
- **Performance disaster.** For every `get_mask()` call, it potentially clones the full parser state and runs `commit_bytes()` for hundreds of candidate tokens.
- **Architectural regression.** The whole point of the DWA precompute pipeline is to avoid per-token probing. This "fix" regresses to the naive approach exactly where it shouldn't.
- **It masks the real bug.** The underlying issue in the DWA weight computation or `possible_matches` construction still exists — this just papers over it.

**Lesson:** When the precompute pipeline under-approximates, the fix must be in the precompute pipeline, not a runtime fallback that defeats the purpose of precomputation.

### Empirical behavior matrix

Tested all four systems with **single Lark grammar** (`STRING_CHAR: /[^\x00-\x1F"\\]/`), after prefix `"`, single byte acceptance:

| Byte | Lark | sep1 | GT(fixed) | GT(old) | xgrammar |
|---|---|---|---|---|---|
| 0x41 (A) | accept | accept | accept | accept | accept |
| 0x7F (DEL) | accept | accept | accept | accept | accept |
| 0x80 (continuation) | accept | reject | reject | accept | reject |
| 0xC0 (overlong lead) | accept | reject | reject | accept | reject |
| 0xC1 (overlong lead) | accept | reject | reject | accept | reject |
| 0xC2 (valid 2-byte lead) | accept | accept | accept | accept | accept |
| 0xDF (valid 2-byte lead) | accept | accept | accept | accept | accept |
| 0xE0 (valid 3-byte lead) | accept | accept | accept | accept | **reject** |
| 0xF0 (valid 4-byte lead) | accept | accept | accept | accept | **reject** |
| 0xF4 (last valid 4-byte lead) | accept | accept | accept | accept | **reject** |
| 0xF5 (invalid lead) | accept | reject | reject | accept | reject |
| 0xFF (always invalid) | accept | reject | reject | accept | reject |

Key findings:
1. **Lark accepts everything ≥0x20** — `surrogateescape` artifact, not deliberate
2. **sep1 and GT(fixed) are identical** — lark_bridge fix works correctly
3. **GT(old) matches Lark** — raw byte passthrough without UTF-8 awareness
4. **xgrammar rejects multi-byte lead bytes (0xE0, 0xF0, 0xF4)** — requires complete UTF-8 characters per token, no cross-token splitting

### Theoretical equivalence analysis

Human asked: given a SINGLE Lark grammar (no changes), are any two frameworks equivalent over ALL grammars and ALL complete inputs?

**Key discoveries during analysis:**
- Sep1 and LLGuidance are **NOT** the same engine. Sep1 uses `_sep1` binary; LLGuidance uses `llguidance` Python package with `LarkCompiler`.
- The three-framework sweep notes: "sep1's `%ignore` WS is not fully supported by llguidance" — confirming they disagree on some grammar constructs.
- xgrammar doesn't parse Lark natively — goes through `lark_to_ebnf()` conversion in `lark_bridge.py`.
- Lark operates on Unicode strings; others operate on bytes.

**Equivalence matrix (general, all grammars, all inputs):**

|  | Lark | Sep1 | xgrammar | LLGuidance |
|---|---|---|---|---|
| Lark | ✓ | ✗ | ✗ | ✗ |
| Sep1 | ✗ | ✓ | ✗ | ✗ |
| xgrammar | ✗ | ✗ | ✓ | ✗ |
| LLGuidance | ✗ | ✗ | ✗ | ✓ |

**No pair is equivalent in general.**

### Architecture redesign decision

Given the fundamental non-equivalence of all frameworks, human decided on a new architecture:

1. **Byte-level semantics everywhere** — pretend Lark does byte-level matching. No UTF-8 awareness in converters.
2. **Framework-agnostic JSON grammar format** — new intermediate representation replacing Lark as interchange.
3. **Schema → JSON grammar converter** — converts JSON schema to the new format.
4. **Four framework-specific converters** (JSON grammar → each framework's native format):
   - sep1 (Lark-like sep1 format)
   - xgrammar (GBNF format, see https://github.com/ggml-org/llama.cpp/blob/master/grammars/README.md)
   - llguidance (Lark-like llguidance format)
   - ground truth checker
5. **All four must behave identically** given the same JSON grammar input.
6. **Sep1 `utf8: true` flag must be removed** — charclasses should NOT get UTF-8 expansion at the sep1 level.
7. **If UTF-8 behavior needed for JSON schema, encode it in the grammar JSON**, not in the converters.

### Key insight

The fundamental problem was: one Lark grammar, four different interpretations. Solution: don't share a grammar format. Instead, share a semantic representation (JSON grammar) and convert to each framework's native format with per-framework converters that ensure identical behavior.
